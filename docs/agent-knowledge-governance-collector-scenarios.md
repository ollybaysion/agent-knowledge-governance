# akg 수집 시나리오 4종 — 시퀀스와 설계 검증 (v0.1)

- 대상 시스템: agent-knowledge-governance (akg), `origin/main @ a2a5337`
- 상위 문서: `agent-knowledge-governance-design.md` v0.4.7,
  `agent-knowledge-governance-collector-design.md` v0.2
- 이 문서의 목적: **"코퍼스를 어떻게 채우는가"를 시나리오로 분해하고, 각 시나리오가
  현 코드에서 실제로 동작하는지 검증한다.** 검증 결과는 수집기 설계 v0.3의 입력이다.
- 검증 방법: 주장별로 코드를 직접 읽어 `file:line` 근거를 남겼다(부록 A). 추정으로
  남은 항목은 그렇게 표시했다.

---

## 0. 한 줄 요약

수집기 설계 v0.2는 **"위키 산문 → 격리 → 활성화 → 주입"이라는 단 하나의 경로**를 다룬다.
그런데 실제 도입에 필요한 시나리오 4칸 중 그 경로에 해당하는 건 1칸뿐이고, **나머지 3칸은
전부 같은 병목 하나에 걸린다 — 문서를 대량으로 만들 경로가 없다.**

그리고 v0.2가 전제한 `quarantined` 상태는 **코드에 존재하지 않는다**(§3.1-9). 즉 "일단
들여놓고 나중에 고른다"는 설계의 뼈대 자체가 아직 없다.

## 1. 시나리오 분해

코퍼스 생애는 두 단계로 갈린다. 사용자 제시 프레임을 그대로 쓴다.

| 단계 | 시나리오 | 원천 | 착지 타입 | 생산자 |
| --- | --- | --- | --- | --- |
| **1. 부트스트랩** | **S1** | Oracle `describe_table` | `db-schema` | 기계(무정제) |
| **1. 부트스트랩** | **S2** | 메시지 예시 | `msg-format` | LLM |
| **2. 정상 운영** | **S3** | 사내 문서 크롤 | 기존 문서의 슬롯 | LLM 분석 |
| **2. 정상 운영** | **S4** | 위키 산문 | `unclassified` | 원문 그대로 |

두 단계는 **정책이 서로 반대**다. 이게 이 문서의 핵심 주장이다.

| 축 | 부트스트랩(S1·S2) | 정상 운영(S3·S4) |
| --- | --- | --- |
| 규모 | 한 번에 전량(수백) | 변경분만(수 건) |
| 검토 | 건별 검토 불가능 | 건별 검토가 유일 방어 |
| 내용 | 팩트만, 슬롯은 비움 | 슬롯 값이 본체 |
| 주입 가치 | 낮음(뼈대) | 높음(그래서 위험) |
| 실패 모드 | 못 만든다 | 큐가 넘친다 |

v0.2는 오른쪽 열 하나만 보고 상한(C0 200개)·격리(C2)·건별 검토(C7)를 설계했다.
왼쪽 열에 그대로 적용하면 부트스트랩은 **실행 자체가 불가능**하다.

  │                      ├───────────────────────►│                │
  │                      │◄───────────────────────┤ 테이블 목록     │
  │                      │ describe_table × N     │                │
  │                      ├───────────────────────►│                │
  │                      │◄───────────────────────┤ 컬럼·타입·코멘트 │
  │                      │                        │                │
  │                      │ 구조 슬롯 채움          │                │
  │                      │ 의미 슬롯 = scaffold    │                │
  │                      │                        │                │

끊기는 건 오른쪽 — akg 서버로 들여보내는 경로다.

## 3. 검증 — 각 시나리오는 현 코드에서 동작하는가

### 3.1 공통: 문서를 만드는 경로

| # | 확인 항목 | 결과 | 근거 |
| --- | --- | --- | --- |
| 1 | 신규 문서 생성 라우트 | `POST /api/docs/:type` **단 하나** | `server/routes/docs.mjs:91` |
| 2 | 그 라우트의 역할 | **`editor` 전용** — agent 토큰 403 | `docs.mjs:93`, `server/auth.mjs:10-11`, `test/server/auth.test.mjs:86-88` |
| 3 | 배치 생성 | **없음.** 요청당 문서 1개, 중복 시 409 | `docs.mjs:99,133-135` |
| 4 | 다중 문서 1커밋 | **없음.** `commitFiles`는 다중 파일을 지원하나 서로 다른 신규 문서에 쓰인 적 없음 | `server/store.mjs:148` 사용처 전수 |

**9번이 이 검증에서 가장 무거운 사실이다.** 수집기 설계 v0.2의 뼈대(격리 착지, 활성화
관문, `deactivate`)가 통째로 미구현이며, 지금은 **문서를 만들면 곧바로 `active`**다.

### 3.2 S1 — 판정: 스키마는 통과, 경로에서 막힘

#### 동작하는 것

- **카탈로그만으로 스키마 유효한 문서를 만들 수 있다.** `purpose`·`columnDescs`를
  `{text: null, tier: "scaffold"}`로 두면 검증을 통과한다(`schemas/common/tiered-value.v1.schema.json:22-34`).
  `columnDescs`는 `minProperties`가 없어 `{}`도 합법이다. 테스트 픽스처가 정확히 이 모양이다
  (`test/server/routes-docs.test.mjs:35-60`).
- `id`는 `lower(table)`로 기계 유도(`src/envelope.mjs:66-75`, owner는 id에 관여하지 않는
  순수 속성), `keywords`는 테이블명을
  기본값으로 쓸 수 있다(`src/migrate/db-schema.mjs:196-199`가 그렇게 한다).
- 즉 **"무정제 수집"은 스키마 차원에서 완전히 정상적인 문서다.** 사람이 지어낼 필수 필드는
  하나도 없다.

#### 막히는 것

1. **생성이 editor 전용이라 자동화가 못 들어간다**(§3.1-2). 스크립트가 사람 토큰을 들고
   도는 방식은 감사 기록이 사람 이름으로 남아 팩트/슬롯 분리 원칙과 충돌한다.
2. **테이블 500개 = HTTP 요청 500번 + 커밋 500개.** 직렬 큐(`server/queue.mjs`)를 500번
   점유하므로 그동안 다른 모든 쓰기가 밀린다.
3. **만드는 즉시 `active`가 되어 인덱스에 들어간다**(§3.1-9, `server/render-store.mjs:22-38`).
   무정제 문서 500개의 키워드 500개가 **곧바로 주입 후보**가 된다.
4. 그런데 주입 슬롯은 턴당 2개이고(`keyword-docs.mjs:226`), 문서당 1200자
   (`:227`)인데 `UserPromptSubmit` 예산은 1500자(`budget.mjs:10`)다 —
   **2 × 1200 = 2400 > 1500이라 실제로는 2개가 다 들어가지도 않는다.** 실효 제한자는
   프로바이더가 아니라 예산 절단이다.
5. **누가 그 슬롯을 이기는지가 비결정적이다.** 소비 측은 먼저 나온 엔트리가 키워드를
   선점하고(`keyword-docs.mjs:197-206`), 그 순서는 상류 `compileIndex`가
   `listIds` = **정렬 없는 `readdirSync`**(`server/store.mjs:127-133`)로 만든 배열 순서다.
   수집기 설계 CS8은 이를 "파일명 알파벳순"이라 썼는데 **부정확하다 — 파일시스템
   readdir 순서이므로 알파벳순보다 나쁘다(재현 보장 없음).**

**판정**: S1은 **"만들 수는 있지만, 만드는 순간 주입 코퍼스가 오염된다."**
`quarantined`가 없으므로 "일단 넣고 나중에 고른다"가 성립하지 않는다.

### 3.3 S2 — 판정: 표현할 자료구조가 없다

**전제 확인**: msg-format 필수 필드 중 **`direction`, `fields[].name`, `fields[].required`
셋은 단일 메시지 예시에서 기계적으로 도출 불가**다(`schemas/msg-format/v1.schema.json:8,22-23`).

- `direction`(`host->equipment` / `equipment->host`)은 메시지 본문에 발신·수신 정보가 없다.
- `required`는 예시가 **여러 개**여야 판정된다.
- 기존 `src/migrate/msg-format.mjs:18-44`도 **사람이 이미 쓴 md 표**를 파싱할 뿐, 원시
  예시에서 뽑는 코드 경로는 존재하지 않는다.

즉 S2는 LLM이 필수이고, 이는 수집기 설계 **C5("LLM 분류기를 수집 경로에 넣지 않는다")와
정면 충돌**한다.

**더 큰 문제 — 우회로도 없다.** "사람이 빈 문서를 만들고 LLM은 propose로 채운다"는
2단 분해를 검토했으나 성립하지 않는다:

- `propose`가 쓸 수 있는 건 **티어드 값 슬롯뿐**이다(`server/routes/proposals.mjs:129-135`의
  `setSlot`).
- 그런데 S2의 핵심 정보인 `fields[].name`·`type`·`required`·`direction`은 **슬롯이 아니라
  팩트 필드**다. 티어드 값은 `purpose`와 `fields[].desc`뿐이다.
- 팩트 필드를 채우는 경로는 `PUT`(editor 수동)과 `catalog-push`(db-schema 전용)뿐이고,
  **msg-format에는 catalog-push 대응물이 없다.**

**판정**: S2는 현 API 조합으로 **불가능**하다. msg-format 부트스트랩은 지금으로선
**사람이 대시보드에서 전량 타이핑**하는 것 외에 경로가 없다.

### 3.4 S3 — 판정: 동작하지만 채택률에 반비례해 무너진다

**동작하는 것**: 제출은 agent 토큰으로 가능하고(`proposals.mjs:19`), 채택은 editor
(`:91`), 승격은 approver(`promote.mjs:51`)로 3단이 이미 선다. 채택 티어는 `inferred`
하드코딩(`proposals.mjs:129-135`)이라 D4도 지켜진다.

#### 무너지는 것

1. **중복 제거가 LLM 앞에서 무력하다.** 키가
   `sha256(JSON.stringify({type, id, slots}))`(`proposals.mjs:33-35`)라 **제안 텍스트가
   한 글자만 달라도 다른 해시**다. 같은 슬롯인지 비교하는 조회는 코드 어디에도 없다 —
   주소 기준 lookup이 없다. 결과: 같은 슬롯에 대해 **매 실행마다 pending이 하나씩 늘어난다.**
2. **상한도 만료도 GC도 없다.** pending은 채택·기각 전까지 영구히 파일로 남는다.
3. **제출마다 pending 전량을 훑는다** — `listIds`(`readdirSync`) + 파일별 `readJson`
   (`proposals.mjs:40-47`). 이게 **단일 직렬 큐 안**에서 돈다(`server/queue.mjs`).
4. **evidence를 제출 시엔 검사하지 않고 채택 시에만 검사한다**(`proposals.mjs:122-128`).
   근거 없는 LLM 제안이 밤새 쌓였다가 사람이 채택하려는 순간 400으로 튕긴다.
5. **인증 성공 요청에는 레이트 리밋이 없다**(`app.mjs:73-79,94`의 카운터는 인증 **실패**만
   센다).

6. **원천이 사람 산문이면 LLM에 먹이는 행위 자체가 상위 §11 S3(문서 = 프롬프트 주입
   채널)의 표면이다.** v0.2는 "외부 문서가 에이전트 사정권에 들어오는 것"을 위협으로
   봤는데, S3은 그것을 파이프라인의 정상 동작으로 만든다. 분석 LLM의 격리 수준은 미설계다.

정량화: 야간 300슬롯 × 30일 무채택이면 pending 약 9,000건 + 커밋 9,000개다. 먼저 무너지는
건 디스크가 아니라 **누적 제곱(n²/2)의 파일 읽기가 직렬 큐를 점유해 다른 모든 쓰기를
밀어내는 것**이다.

**판정**: **동작하지만 가드 없이 켜면 안 된다.** 필요한 가드 3종은 §5에 정리했다.

### 3.5 S4 — 판정: P0 전체가 선행 조건

v0.2 설계가 다루는 칸이지만 전제가 전부 미구현이다. 추가로 **격리가 배포를 막지 못한다는
설계 주장은 코드로 재확인됐다**:

- `docWrites`는 status와 무관하게 `.md`를 쓴다(`server/render-store.mjs:53-81`).
- `compileIndex`만 `active` 필터를 건다(`:22-38`) — 즉 인덱스에서만 빠지고 파일은 남는다.
- `/api/bundle`은 `tar -czf - -C storeDir rendered`로 **디렉토리를 통째로** 만다
  (`server/routes/misc.mjs:83`).
- `syncMirror`는 status를 보지 않고 그대로 푼다(`src/mirror/sync.mjs`).

→ **비활성 문서의 렌더 결과물이 전 사용자 미러에 물리적으로 도달한다.** 설계 C2의 진단이 맞다.

부수 확인 2건:

- `spawnSync`에 `maxBuffer` 미지정(기본 1 MiB) — 실측 결과 초과 시 `SIGTERM` + `ENOBUFS`,
  `r.status === null`이 되어 `/api/bundle`이 **500 `bundle_failed`**를 반환한다.
  손상된 tar를 조용히 서빙하지는 않지만, `rendered/`가 gzip 1 MiB를 넘는 순간 **영구 500**이다.
- `assertKnownTopLevel` 실패 시 CLI는 **exit 0**이다(`bin/akg.mjs:128-134`) — 401만 exit 1.
  의도된 fail-open이지만, 포맷 불일치와 서버 다운을 구분하지 못해 **미러가 무증상으로
  동결**된다.

## 4. 공통 병목 3개

시나리오 4칸이 각자 다른 이유로 막히는 것처럼 보이지만, 실은 셋으로 수렴한다.

**병목 1 — 대량 생성 경로 부재.**
단건 editor POST가 유일하다. S1·S2·(그리고 기존 md 적재)가 전부 여기 걸린다.

**병목 2 — 비활성 착지 상태 부재.**
`status`가 `active`/`archived`뿐이라 "코퍼스에는 있으나 주입되지 않는" 중간 상태가 없다.
이게 없으면 부트스트랩은 **주입 코퍼스를 직접 오염시키는 행위**가 된다. `archived`로
대신하려 해도 렌더·미러가 status를 안 보므로 배포는 그대로 일어난다(§3.5).

**병목 3 — 팩트 필드 쓰기 경로가 타입마다 다르다.**
db-schema만 `catalog-push`가 있고 msg-format·domain-skill·unclassified엔 대응물이 없다.
그래서 "기계가 아는 사실"을 넣는 방법이 타입마다 다르거나 아예 없다.

## 5. 수집기 설계 v0.3 개정안

### R1. 수집기를 두 모드로 분리한다

v0.2의 단일 파이프라인을 **부트스트랩 임포터**와 **증분 수집기**로 나눈다.
상한·검토·격리 정책이 반대이므로 한 정책으로 덮을 수 없다.

| 축 | 임포터(S1·S2·기존 md) | 증분 수집기(S3·S4) |
| --- | --- | --- |
| 상한 | 없음(전량이 목적) | C0 유지(실행당 200) |
| 착지 | 비활성 + 슬롯 비움 | 비활성 + 배치 검토 |
| 검토 단위 | **문서 아님 — 활성화만** | 문서 단위 채택 |
| 실행 주체 | 사람이 1회 기동 | 스케줄러 주기 |

### R2. 비활성 상태를 먼저 만든다 (P0 최우선)

`status` enum에 비활성 값을 추가하고, **렌더·번들·미러 전 계층이 그 값을 존중**하게 한다.
이름은 v0.2의 `quarantined`를 쓰되, 부트스트랩 문서에는 "격리(의심)"보다 "미활성(뼈대)"이
맞으므로 **의미 축을 상태가 아니라 `source` 유무로 구분**할 것을 제안한다.

이것이 R1~R5 전부의 선행 조건이다. 지금은 만드는 즉시 주입 대상이 되므로 **어떤 대량
적재도 안전하지 않다.**

### R3. 정형 타입 대량 생성 경로를 만든다

두 안 중 택일:

- **(a) 배치 생성 라우트** — `POST /api/docs/:type/batch`(editor), N개를 **1커밋**으로.
  512 KiB 상한 때문에 청크가 필요하다(수집기 설계 C3의 청크 제출과 같은 문제).
- **(b) 서버측 임포터 CLI** — `akg import --from <dir>`가 서버 프로세스 안에서 store에
  직접 쓴다. HTTP 상한을 우회하지만 **역할 강제가 `preHandler`에만 있다는 C6의 함정**에
  그대로 빠지므로, 루프백 HTTP 규율(C6)과 모순되지 않게 설계해야 한다.

현 코드 구조(직렬 큐 + `commitFiles` 다중 파일 지원)를 보면 **(a)가 자연스럽다** —
`commitFiles`는 이미 다중 파일을 받는다(`server/store.mjs:148`).

### R4. 팩트 쓰기를 일반화한다

`catalog-push`(db-schema 전용)를 **`PUT /api/docs/:type/:id/facts`로 일반화**하고,
없으면 **생성**하도록 upsert 의미를 준다. 이 하나로 S1(카탈로그 팩트)과 S2(메시지
구조 팩트)가 같은 경로를 쓴다. 슬롯은 절대 건드리지 않는다는 기존 catalog-push의 규율을
그대로 승계한다.

### R5. S3 가드 3종

1. **주소 단위 upsert.** dedup 키를 `(type, docId, slotAddress, submittedBy)`로 바꾸고,
   같은 주소의 pending이 있으면 **교체**한다. 텍스트 해시는 "변경 없음" 판정에만 쓴다.
2. **pending 상한과 만료.** 제출자별 상한 + N일 경과 자동 만료(또는 대시보드 노후 집계).
   수집기 설계 CS6의 격리 노화 관리와 같은 규율을 제안 큐에도 적용한다.
3. **제출 시 evidence 검증.** 채택 시점으로 미루면 실패가 사람의 시간에 청구된다.

### R6. C5(LLM 배제)의 적용 범위를 좁힌다

"LLM을 넣지 않는다"는 **수집 경로**(원문을 코퍼스로 들이는 경로)에만 적용하고,
**제안 경로**에는 적용하지 않는다. 근거: claude-hooks `db-schema-propose-codebase`가
이미 LLM 생산자로 정상 운영되고 있으며, 그 산출물은 사람 채택을 거치므로 결정성이
필요한 지점이 아니다. S2·S3은 이 경계 재정의 없이는 설계상 금지된 동작이 된다.

### R7. 기존 생산자를 재발명하지 않는다

S1의 왼쪽 절반(describe → 구조 슬롯 채움 → 의미 슬롯 scaffold)은 claude-hooks
`db-schema-docs` 스킬로 **이미 동작한다**. S3도 `db-schema-propose-codebase` +
`db-schema-apply`가 원천만 다를 뿐 같은 모양이다.

따라서 수집기가 어댑터·JSONL·배치를 새로 발명하기 전에, **기존 스킬의 출력을 akg에
쓰는 경로**(R3·R4)를 먼저 만드는 것이 비용 대비 효과가 크다. akg Phase 3에서 "ch 스킬을
akg CLI로 개조"를 범위 밖으로 뺐지만, 부트스트랩 관점에서는 그것이 **핵심 경로**다.

### R8. 문구 정정 2건

- **CS8**: "인덱스 순서 = 파일명 알파벳순" → **정렬 없는 `readdirSync` 순서**로 정정.
  알파벳순보다 나쁘다(재현 보장 없음). 명시적 정렬 키 도입 근거는 오히려 강해진다.
- **§0.1**: "턴당 2개 × 1200자" 옆에 **"단, `UserPromptSubmit` 예산이 1500자라 2개가 다
  들어가지 않는다"**를 병기. 슬롯 희소성 논거(C0)가 더 강해진다.

## 7. 열린 질문

1. **부트스트랩 문서를 누가 활성화하는가.** 500개를 approver가 건별로 켜는 건 불가능하다.
   "타입 단위 일괄 활성화"를 허용할지, 아니면 **활성화 없이도 조회는 되는 경로**(대시보드
   검색·B모드 명시 fetch)를 여는지 — 후자는 v0.2 §11-5의 열린 질문과 같은 지점이다.
2. **자동화 주체의 역할.** 임포터가 agent 토큰을 쓸 수 있게 할지(D6 완화), 아니면
   전용 `importer` 역할을 신설할지.
3. **감사 저자.** 기계가 만든 500개 문서의 `by`를 무엇으로 남길지(`import:<runId>` 제안).
4. **msg-format 생산자를 누가 만드는가.** `db-schema-docs`의 대응물이 없다 — 신규 스킬인지,
   forge 계열인지, akg 자체 기능인지 미정.
5. **S3 분석 LLM의 격리 수준.** 외부 산문을 먹이는 경로의 프롬프트 주입 대응이 미설계.
