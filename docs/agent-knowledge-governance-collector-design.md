- 대상 시스템: agent-knowledge-governance (akg), main @ `a731a4e` (Phase 0~3 구현 완료)

## 0. 한 줄 요약

수집기는 **이미 사내에 존재하는 문서(레포 파일·위키)를 akg 코퍼스로 끌어오는 인입구**다.
어댑터가 원천을 긁고, 순수 코어가 정규화·차분하고, 착지 계층이 **검토 대기 상태로만**
문서를 만든다. 수집기는 사람의 검토를 대체하지 않으며, **어떤 경로로도 주입되는
문서를 스스로 만들지 못한다**.

### 0.1 규모의 진실 — 이 설계를 지배하는 실측치

소비 측(claude-hooks `keyword-docs.mjs`·`budget.mjs`) 실측:

| 값 | 수치 |
| --- | --- |
| 한 턴에 주입되는 문서 수 | **2개**(`maxDocs`) |
| 문서당 주입 문자 수 | **1200자**(`maxCharsEach`) |
| `UserPromptSubmit` 전체 예산 | **1500자** |

**주입 슬롯은 턴당 2개다 — 단, 2개가 다 들어가지도 않는다.** `2 × 1200 = 2400`자는
`UserPromptSubmit` 예산 1500자를 넘고, 그 1500자는 keyword-docs 전용이 아니라 그 이벤트의
**모든 프로바이더가 나눠 쓰는 총액**이다(`budget.mjs`의 `budgetFragments`가 우선순위대로
담다 자른다). 즉 실효 제한자는 프로바이더의 `maxDocs`가 아니라 **예산 절단**이며, 슬롯은
명목 수치보다 더 희소하다. 위키 3000페이지를 수집해 전부 활성화해도 한 턴에 들어가는
건 문서 2개에 못 미친다. 반면 활성화 1건마다 approver는 프롬프트 주입 책임을 개인적으로
진다(상위 §11 S3).
**비용은 건당 선형, 수익은 수십 건에서 포화** — v0.1대로면 합리적 approver의 최적 전략은
"아무것도 활성화하지 않기"였다. 격리 코퍼스가 죽은 데이터가 되는 건 게으름이 아니라
인센티브 구조의 정상 귀결이었다.

따라서 이 설계의 목표는 "**많이 긁어오기**"가 아니라 "**희소한 주입 슬롯에 무엇이
들어갈 자격이 있는지 가리는 것**"이다. 대량 크롤은 지원하지 않는다(C0).

추가 실측: **한글 키워드는 발화하지 않는다.** 기본 `word` 모드는 `\b키워드\b`인데 JS `\b`는
ASCII `\w` 기준이라 한글 양옆에서 앵커되지 않는다(`"센서"` → word `false`, exact `false`,
substring만 `true`; 영문 `"sensor"`는 `true`). 봉투 스키마 패턴 `^[a-z0-9_. -]+$`가 한글을
애초에 거부하므로 실사고는 안 나지만, **한국어 위키에서 키워드를 고르라고 하면 사람은
억지 영단어(`data`·`report`·`config`)를 짜내게 되고 그게 광역 키워드 오탐 폭발 경로**가
된다 → C7의 근거.

### 0.2 v0.1에서 깨진 것 (검증 결과)

| v0.1 주장 | 실제 | 반영 |
| --- | --- | --- |
| "격리는 주입을 코드 변경 0으로 차단" | `/api/bundle`이 `rendered/`를 **status 무관 전량 tar** → 격리본이 전 사용자 미러에 배포 | C2 재작성 |

## 1. 문제와 범위

### 1.1 왜 필요한가

akg 코퍼스에 문서가 들어오는 경로는 현재 4개뿐이고, 전부 "사람이 하나씩" 계열이다.

| 경로 | 무엇이 들어오나 | 한계 |
| --- | --- | --- |
| `migrate-md` (Phase 0) | 기존 md 4개 | 일회성 마이그레이션 도구, 재실행 전제 없음 |
| `PUT .../catalog` (catalog-push) | DB 카탈로그 **팩트** | db-schema 타입 전용, 문서를 **만들지는** 못함(`404 not_found`) |
| `POST /api/proposals` (propose) | 기존 문서의 **슬롯 값** | 대상 문서가 이미 있어야 함 |
| 대시보드 편집 | 사람이 친 것 | 규모가 안 남 |

사내 도입 시 실제 도메인 지식의 상당수는 이미 **레포의 md/README와 위키**에 있다.
akg가 "도메인 지식 SSoT 관리 툴"이 되려면 그 지식을 거버넌스 안으로 들여오는
반복 가능한 인입구가 있어야 한다. 그게 수집기다.

### 1.2 목표

- 레포 파일·사내 위키 등 외부 원천에서 **지정한 범위의 문서를 반복적으로(멱등하게) 수집**
- 수집물을 akg 문서로 착지시키되, **주입 표면은 사람만 열고, 열린 뒤에도 계속 지킨다**(C2)
- 재수집 시 **원천 변경분만** 반영하고 **사람이 만든 것은 절대 덮지 않는다**(C9)
- 원천에서 사라지거나 이동한 문서를 감지해 보고한다(자동 삭제 없음)
- 러너 2종(사람이 돌리는 CLI, 서버 스케줄러)이 **같은 수집 코어**를 쓴다
- 잘못 들어온 것을 **배치 단위로 되돌릴 수 있다**(C4-b)

### 1.3 비목표

- **대량 크롤.** 위키 전체를 긁어 미러링하는 것은 이 시스템의 목적이 아니다(C0).
- **문서 분류 자동 판단.** 위키 페이지가 db-schema인지 msg-format인지 수집기는 모른다(C5).
  LLM 분류기를 수집 경로에 넣지 않는다 — 결정성이 깨지고, 오분류는 스키마 검증 실패로
  남아 큐를 오염시킨다.
- **원천 시스템별 인증·HTML 변환**을 akg 코어가 떠안는 것(C1).
- 전문 검색·유사도 중복 판정. 동일성은 **정확 해시 일치**로만 본다(C8의 `rename` 포함).
- 기존 `POST /api/proposals`(슬롯 제안) 계약 변경(C3).

## 2. 현 상태의 실측 — 수집기가 부딪히는 벽

설계는 이 코드 사실들 위에 서 있다. **v0.1의 §2-2는 오독이었으므로 정정한다.**

1. **신규 문서 인입 경로 부재.** `proposals.mjs`의 adopt는 대상 문서가 없으면
   `404 target_doc_not_found`. 제안은 **기존 문서의 슬롯만** 표현할 수 있다.
2. **`unclassified/v1`은 스키마만 있고 서버가 안 쓴다.** `docs.mjs`의 `DOC_TYPES`는 3종,
   `render-store.mjs`의 `INDEXED_TYPES`는 2종이라 렌더도 인덱스도 없다. 그리고
   **`store.mjs:131`의 `listIds`는 `.meta.json`을 결과에서 배제**하며
   `test/server/store.test.mjs:81`이 `listIds(dir,"unclassified") === []`로 그 동작을
   못박고 있다 — 자리 예약이 아니라 **배제**다. 전용 리스터가 필요하다(§5.4).
3. **주입 차단 레버는 봉투에 있으나 배포는 못 막는다.** `compileIndex`는 `active`가
   아닌 문서를 인덱스에서 버리지만, `docWrites`는 status와 무관하게 md를 쓰고
   `/api/bundle`(`misc.mjs:83`)은 `tar -czf - -C storeDir rendered`로 **`rendered/`를
   통째로** 만다 → C2가 이를 정면으로 다뤄야 한다.
4. **제안 큐는 대량 유입에 취약하다.** 제출 시 dedup이 `proposals/pending` 전체를
   `readJson`으로 훑는다(O(n) per submit) → C3의 근거.
5. **`docWrites`가 매 문서 쓰기마다 인덱스를 전량 재컴파일한다**(`listIds` + 전체
   `readJson`). N건 일괄 채택 = O(N²) 동기 파일 읽기 → CS9.
6. **전역 `bodyLimit`은 512KB**(`app.mjs:30`, S13) → CS5 재산정의 근거.
7. **`spawnSync`가 `maxBuffer` 미지정(기본 1MB)** — `git.mjs:9`·`misc.mjs:83` 모두.
   `rendered/`가 gzip 1MB를 넘는 순간 `/api/bundle`이 `ENOBUFS`로 **영구 500**이고,
   `sync`는 fail-open이라 **전 사용자 미러가 조용히 얼어붙는다** → §13 선행 수정.
8. **`src/mirror/sync.mjs`의 `ALLOWED_TYPE_DIRS`에 `unclassified`가 없고**
   `assertKnownTopLevel`이 모르는 최상위 항목에 throw한다. 서버가 그 디렉토리를
   내보내는 순간 배포된 CLI의 sync가 **전부 실패**한다(exit 0이라 무증상) → CS11.

## 3. 아키텍처 개요

```text
   ── 신뢰 경계 ──┐
 ┌─────────────┐ │  JSONL     ┌──────────────┐  청크 제출   ┌──────────┐
 │ 어댑터       │ │ (stdout)  │ 수집 코어      │ ──────────► │ 착지     │
 │ (원천별·외부) │─┼──────────►│ (순수 함수)    │             │ 클라이언트│
 └─────────────┘ │            └──────────────┘             └────┬─────┘
   repo-fs(내장) │             정규화 → 지문 →                    │ HTTP
   wiki-*(사내)  │             제안 상태와 차분                     ▼
                 │                                        ┌───────────────┐
   ┌──────── 러너 ────────┐                                │  akg 서버      │
   │ akg collect (editor)  │                               │ /api/intake/* │
   │ 스케줄러 (루프백 HTTP,  │                               └───────┬───────┘
   │          agent 토큰)   │                                       │ 사람 검토
   └──────────────────────┘                                        ▼
                                          store/unclassified/<id>.md  (quarantined)
                                                        │ activate (approver, 키워드 확정)
                                                        ▼
                                          rendered/unclassified/  →  번들 →  주입
```

**어댑터는 신뢰 경계 밖이다**(CS1). 3층 분리의 이유는 변경 이유가 다르기 때문이다 —
어댑터는 원천 시스템이 바뀔 때, 코어는 차분 규칙이 바뀔 때, 착지는 거버넌스 정책이
바뀔 때 바뀐다.

## 4. 핵심 설계 결정

### C0. 수집은 **지정 수집**이지 전량 크롤이 아니다 (신설)

§0.1의 실측이 이 결정을 강제한다. 주입 슬롯이 턴당 2개인 싱크에 3000페이지를 밀어넣는
것은 어느 층에서도 이득이 없고, 규모에 선형·제곱으로 반응하는 결함
(요청 상한·큐 누적·인덱스 재컴파일·번들 배포·키워드 충돌·id 충돌·취소 비용)을 전부
활성화시킨다.

**결정**: 수집 실행 1회의 항목 수를 **기본 200개로 제한**하고, 초과 시 배치를 거부하며
"범위를 좁히라"고 안내한다(어댑터의 경로·라벨·스페이스 필터를 쓰라는 뜻). 상한은
설정으로 올릴 수 있으나 **기본값이 정책**이다.

akg는 위키의 복제본이 아니라 **SSoT**다. "긁을 수 있는 만큼 긁는다"는 목표 자체가 틀렸다.

> 규모 방어는 상한 하나에만 의존하지 않는다. CS9(인덱스 재컴파일 O(N)화)·CS5(청크 제출)·
> C3(항목 단위 저장)은 상한을 올려도 견디도록 별도로 둔다.

### C1. 어댑터는 서브프로세스 계약(JSONL) — 내장은 `repo-fs` 하나뿐

akg는 **런타임 의존성 0**(ajv조차 수제 `src/validate.mjs`)이 규율이다. Confluence
SDK·HTML→md 변환기를 코어에 끌어들이면 이 규율이 깨지고, 사내 위키가 Confluence가
아닐 경우 통째로 쓸모없어진다.

**결정**: 어댑터는 akg 프로세스 밖의 실행 파일이다. 계약은 **stdout에 JSONL 한 줄 =
수집 항목 하나**뿐(§5.1). akg는 그 JSONL을 검증·차분·착지만 한다.

- 내장 어댑터는 `repo-fs` 하나(글롭 + git 메타).
- 사내 위키는 각 팀이 얇은 스크립트를 쓴다. 인증·페이지네이션·HTML 변환은 **어댑터 책임**.
- 결과: 새 원천 추가에 akg 릴리스가 필요 없다.

**어댑터 실행은 그 자체가 공격면이다** — CS1에서 상세히 규율한다. §3 그림처럼 어댑터는
**신뢰 경계 밖**이며, "akg는 검증·차분·착지만 한다"는 문장은 *데이터*에 대해서만 참이고
*실행*에 대해서는 거짓이다.

### C2. 격리(`quarantined`)는 주입뿐 아니라 **배포**를 막는다

상위 §11 S3(문서 = 프롬프트 주입 배포 채널)이 이 프로젝트 고유의 공급망 위협이다.
수집기는 사람 관문 없이 외부 텍스트를 끌어오는 장치라 이 위협을 정면으로 키운다.

**결정 (a) — 상태**: 봉투 `status` enum을 **`active | quarantined | archived`**로 확장한다.
수집기가 만든 문서는 예외 없이 `quarantined`로 착지한다.

**결정 (b) — 배포 차단**: `status !== "active"`인 문서는 **`rendered/`에 아예 쓰지 않는다.**
activate 시점에 렌더/복사하고, 재격리·아카이브 시 삭제한다. 추가로 `/api/bundle`을
디렉토리 통짜 tar에서 **`index.json`이 참조하는 파일 화이트리스트 tar**로 바꿔
status와 배포를 구조적으로 묶는다(누락 시 재발하는 종류의 결함이므로 구조로 막는다).

> v0.1은 "`compileIndex`가 active만 통과시키므로 주입은 코드 변경 0으로 차단"이라고
> 썼는데, 차단되는 건 **인덱스 항목**뿐이고 md 파일 자체는 번들로 나간다. 검토 전
> 외부 텍스트가 전 개발자 디스크의 `~/.claude/akg/`에 착지하면 키워드 트리거를 거치지
> 않고도 에이전트의 Read/Glob/grep 사정권에 들어온다. **격리는 "안 보임"이 아니라
> "안 나감"인데, v0.1은 안 나가게 하지 못했다.**

**결정 (c) — 지속적 게이트**: D4는 *매 쓰기* 불변식인데 v0.1의 격리는 *1회* 게이트였다.
active 문서의 상류가 바뀌어 `update`가 채택되면 승인된 주입 표면의 **내용물이 approver
없이 교체**된다. 따라서 **`update`가 `active` 문서를 대상으로 하면 채택과 동시에
`quarantined`로 되돌리고 재활성화(approver)를 요구**한다(§6의 `update-active` 행).

이로써 "**어떤 수집 경로도 active 문서를 만들거나 유지할 수 없다**"가 D4와 같은 급의
불변식이 된다.

**대안 기각**: keywords를 비워 주입 차단 → 스키마 `minItems: 1` 위반(그리고 C7이 이
제약 자체를 조건부로 푼다). `inject: "pointer"`만 허용 → 포인터도 주입 경로다.

### C3. 착지는 **수집 배치(intake batch)** — 항목 단위 저장, 청크 제출

수집은 "한 번에 N건"이 정상이고, 사람의 검토 단위도 **"이번 수집에서 새로 들어온 것들"**
이다. 반면 `proposals`는 슬롯 값 하나를 놓고 문맥을 따지는 큐이며 §2-4의 O(n) dedup
때문에 수집물을 개별 제안으로 밀면 큐가 무너진다.

**결정**: `store/intake/pending/<runId>/` — **수집 실행 1회 = 배치 1건, 항목은 파일 1개씩.**

```text
store/intake/pending/<runId>/
  manifest.json          # 어댑터·시각·stats·상태(open|sealed)
  items/<refHash>.json   # 항목 하나 (본문 제외)
  blobs/<contentHash>    # 본문 (채택 시 참조, 기각 시 커밋 전 폐기)
```

- **제출은 청크**로 받는다(`POST /api/intake/:runId/chunk` ≤ 256KB → `POST .../seal`).
  단일 요청 상한(512KB)에 수집 규모가 직결되던 v0.1 구조를 끊는다.
- **목록 API는 `manifest.json`만 읽는다** — 32MB 배치 N개를 전부 파싱해야 요약이 나오던
  v0.1 구조를 끊는다.
- 항목별 액션:

| 액션 | 의미 | 채택 시 서버 동작 |
| --- | --- | --- |
| `create` | 원천에 새로 생김 | `unclassified` 문서 생성 (`quarantined`) |
| `update` | 원천 본문이 바뀜 | md 본문 교체(사이드카 사람 필드 불변, C9) |
| `update-active` | 위 + 대상이 `active` | 본문 교체 **+ `quarantined`로 강등**(C2-c) |
| `rename` | 이동/개명 (해시 동일·ref 상이) | 기존 문서 유지, `source.ref`만 갱신(C8) |
| `orphan` | 원천에서 사라짐 | **자동 동작 없음** — 보고만(CS3) |
| `conflict` | 상류·사람 편집 동시 변경 | 보고만, 사람이 선택(C9) |

- 기존 `POST /api/proposals`는 **한 줄도 안 바꾼다**.
- 정형 타입 문서의 상류 변경은 `update`가 아니라 **슬롯 제안으로 우회**한다(C9).

### C4. 출처 지문과 **제안 상태 기준선**

**(a) 지문** — 봉투(정확히는 unclassified 사이드카, §5.4)의 `source`:

```jsonc
"source": {
  "adapter": "repo-fs",
  "ref": "fdc-app@docs/sensor-guide.md",   // 원천 내 안정 식별자(페이지 ID 권장 → CS12)
  "contentHash": "sha256:9f1e…",
  "fetchedAt": "2026-07-21T02:10:00Z",
  "runId": "20260721-021000-repo-fs"
}
```

`source`가 **없는 문서는 수집기가 절대 건드리지 않는다.** 사람이 만든 문서와 수집
문서의 경계가 이 필드 하나로 결정된다.

**(b) 차분 기준선은 문서가 아니라 "마지막 제안"이다** — v0.1의 치명적 누락.
v0.1은 멱등 비교 대상을 *착지한 문서*로 뒀는데, **채택 전에는 문서가 없다.** 아무도
채택하지 않으면 야간 스케줄러가 **매일 같은 전량을 다시 제안**한다(30일 = 30배 누적).

따라서 `store/intake/state/<adapter>.json`에 `ref → {contentHash, proposedInRunId}`를
유지하고 **미채택 항목도 여기 기록**한다. 같은 해시면 새 배치에 싣지 않는다.

**(c) supersede는 기본값이며 P0 결정이다**(열린 질문 아님). 새 배치가 같은 `ref`를 담으면
이전 배치의 그 항목을 `superseded`로 잠근다. 이것 없이는 검토자가 백로그를 임의 순서로
처리할 때 **오래된 본문이 최신을 조용히 덮어쓴다**(409도 경고도 없이).

**(d) adopt의 `If-Match`는 배치 rev + 대상 문서 rev 둘 다** 요구한다. v0.1은 배치 rev만
봐서 상위 §11의 S4·S5 규율을 수집 경로만 우회시켰다.

### C4-b. 되돌리기는 채택과 **같은 단위**(배치)여야 한다

채택이 대량 연산인데 취소가 단건 `DELETE`뿐이면 비대칭이다(3000건 취소 = 커밋 3000개).
게다가 S9가 수동 git 개입을 금지하므로 `git revert` 같은 자연스러운 롤백은 **규율상·
코드상 모두 막혀 있다** — 서버가 제공하지 않는 되돌리기는 존재하지 않는다.

**결정**: `POST /api/intake/:runId/revert`(approver)가 해당 runId로 채택된 문서를
**한 커밋으로** 되돌린다(`create`였던 건 삭제, `update`였던 건 `readJsonAtRev`로 이전 본문
복원). 이를 위해 채택 시 `store/intake/adopted/<runId>.json`에 `runId → docId[]` 매핑을
남긴다(감사 검색 문제도 함께 해결 — CS10).

또한 **`active → quarantined` 전이**(`POST .../deactivate`, approver)를 추가한다.
"주입 중지"와 "폐기(`archived`)"는 다른 사건이다.

### C5. 기본 착지 타입은 `unclassified` — 분류는 별도 단계

수집기는 원천 텍스트가 어떤 정형 타입인지 판단할 근거가 없다.

**결정**: 수집물은 전부 `unclassified/v1`로 착지한다. 정형 타입으로의 승격(재분류)은
사람 주도의 별도 동작이며 v1 범위 밖이다(P4).

이 결정은 상위 §12-4("범용 keyword-docs를 허브에 넣을지, 로컬에 남길지 — 초안: 로컬 잔류")를
**"허브에 넣는다"로 뒤집는다**. 그에 따라 §5.4의 구현 부채를 인수한다.

### C6. 러너는 2종이고 권한이 다르다 — **단, 역할은 HTTP 경계에서만 강제된다**

| 러너 | 실행 주체 | 어댑터 실행 위치 | 할 수 있는 것 |
| --- | --- | --- | --- |
| `akg collect` (CLI) | 사람(editor 토큰) | 사용자 머신 | 배치 제출 (+`--direct`, §7.2) |
| 스케줄러 | **agent 토큰** | 서버(화이트리스트, CS1) | **배치 제출만** |

**v0.1의 오류 정정**: "스케줄러가 agent 역할이므로 권한 모델만으로 강제된다"는 거짓이다.
이 서버에서 역할 강제 지점은 **`app.mjs`의 `preHandler` 훅 하나뿐**이고 `request.user`는
거기서만 채워진다. 인프로세스 스케줄러가 `queue.enqueue`+`commitFiles`를 직접 부르면
(기존 라우트가 전부 그렇게 생겼으므로 자연스럽게 그렇게 된다) **agent 제약이 하나도
적용되지 않는다.**

**결정**: 스케줄러는 **자기 agent 토큰으로 루프백 HTTP를 통해서만** 서버에 쓴다
(`127.0.0.1`, S2의 바인딩 규율과 정합). 인프로세스 지름길을 금지한다.

### C7. 키워드는 **채택이 아니라 활성화 시점에** 확정한다

v0.1은 봉투 스키마 `minItems: 1` 때문에 "채택 시 최소 1개 선택 강제"라고 했다. 200건
채택이면 **200번의 키워드 결정**이 채택의 전제가 된다. 실제 행동은 뻔하다 — 후보를 전부
수락하고 일괄 통과. **검토 관문이 형식만 남고 실질이 0이 된다.** 그런데 상위 §11 S3은
"검토 관문이 유일한 실질 방어"라고 못박았다.

**결정**: 격리 문서는 주입되지 않으므로 **키워드가 필요 없다.** unclassified 스키마에
조건부 규칙을 넣어 `status === "quarantined"`일 때 `keywords` `minItems`를 면제하고,
**키워드는 activate 시점에만 강제**한다.

그러면 채택은 "이 페이지를 코퍼스에 넣을까"라는 **예/아니오 한 번**이 되고, 실제 판단
비용(키워드·주입 책임)은 활성화하는 **소수 문서**에만 든다. §0.1의 인센티브 왜곡이
여기서 해소된다.

**활성화 화면은 "무엇을 넣을까"가 아니라 "무엇을 빼고 이걸 넣을까"다.** 슬롯이 희소하다는
사실(턴당 2개)을 UI가 말해야 approver의 판단이 유의미해진다.

### C8. `docId`는 `ref`만의 순수 함수 — 그리고 이동은 해시로 감지한다

v0.1의 "순수 함수" 선언과 "충돌 시 `-2`"는 서로 모순이었다(`-2`는 저장소 상태 의존).
또한 한글 경로는 slug 결과가 빈 문자열이 되고, `docs/a.md`와 `docs.a.md`가 같은 id로
충돌해 **먼저 수집된 쪽이 id를 선점**하는 스쿼팅이 가능했다.

**결정**:

- `docId = slug(ref) + "-" + sha256(ref).slice(0,6)`. 충돌이 **구조적으로 불가능**해지고
  "순수 함수"가 실제로 참이 된다. slug가 비거나 `..`을 포함하면 해시 전용 id로 폴백.
- **서버는 배치의 `docId`를 신뢰하지 않고 `ref`에서 재유도한다**(CS7).
- **`rename` 액션**: `orphan` 후보의 `contentHash`가 `create` 후보와 일치하면 이동으로
  판정해 1건으로 병합한다(구 docId 유지, `source.ref`만 갱신, 키워드·사이드카 전부 보존).
  정확 해시 일치라 §1.3의 "유사도 판정 안 함"을 위반하지 않는다. 이것 없이 위키 대량
  이동이 일어나면 `new 500 / orphaned 500`이 뜨고 검토자는 이동인지 삭제+신규인지
  구분할 수단이 없다.

### C9. 사람이 만진 것은 수집기가 덮지 못한다

팩트/슬롯 분리(상위 §0 온톨로지)의 문서 단위 재현이다.

- unclassified에서 **수집기 소유 = md 본문 + `source`**. 사이드카의 `keywords`·`status`·
  수동 편집분은 **사람 소유**.
- 사람이 본문을 직접 편집하면 사이드카에 `bodyEditedBy`가 남고, 이후 상류 변경은
  `update`가 아니라 **`conflict`로 보고**된다(자동 병합 없음). 3-way 병합은 범위 밖 —
  Phase 1 서버가 이미 문서화한 한계와 같은 선.
- 정형 타입으로 재분류된 문서는 슬롯 제안으로만 접근한다.

### C10. dry-run이 기본값

상위 §6이 "dry-run 문화는 클라이언트 CLI 쪽에 유지"라고 못박았다. `akg collect`는 **기본이
dry-run**이고 제출은 `--submit` 명시가 필요하다.

## 5. 데이터 모델

### 5.1 어댑터 출력 — JSONL 항목 (계약)

```jsonc
{
  "ref": "fdc-app@docs/sensor-guide.md",   // 필수. 안정 식별자(페이지 ID 권장, CS12)
  "title": "센서 가이드",                    // 필수
  "mime": "text/markdown",                  // 필수. v1은 이 값만 허용(CS4)
  "text": "# 센서 가이드\n\n…",              // 필수
  "updatedAt": "2026-07-19T04:00:00Z",      // 선택
  "keywords": ["fdc_sensor"],               // 선택. 후보일 뿐 — 확정은 activate(C7)
  "sourceUrl": "https://wiki/…"             // 선택
}
```

모르는 키는 **거부**(어댑터 오작동의 조용한 통과 금지). 상한은 CS5.

### 5.2 배치 — 매니페스트 + 항목 파일 (§C3)

```jsonc
// manifest.json
{
  "runId": "20260721-021000-repo-fs",
  "adapter": "repo-fs",
  "submittedBy": "collector:nightly",
  "createdAt": "2026-07-21T02:10:00Z",
  "state": "sealed",
  "stats": { "scanned": 412, "new": 7, "changed": 3, "renamed": 1,
             "unchanged": 400, "orphaned": 2, "oversize": 1 }
}
// items/<refHash>.json
{
  "action": "create",
  "ref": "fdc-app@docs/sensor-guide.md",
  "title": "센서 가이드",
  "contentHash": "sha256:9f1e…",
  "blob": "blobs/sha256-9f1e…",
  "keywordCandidates": ["fdc_sensor"],
  "superseded": false
}
```

`docId`는 배치에 싣지 않는다 — 서버가 `ref`에서 재유도한다(C8·CS7).

### 5.3 id·키워드 유도

- `docId` = C8의 순수 함수. 서버가 재유도하고 `validateId`로 재검증한다.
- **키워드는 채택 시 확정하지 않는다**(C7). `keywordCandidates`는 activate 화면에서만 쓰인다.
- 활성화 시 **키워드 충돌 검사**가 선행된다(CS8).

### 5.4 unclassified 타입의 구현 부채 (수집기의 선행 조건)

| 항목 | 현 상태 | 필요한 것 |
| --- | --- | --- |
| 저장 | 없음 | `store/unclassified/<id>.md` + `<id>.meta.json` |
| **스키마** | `additionalProperties:false`, `status` 2값 | **`quarantined` 추가 + `source`·`bodyEditedBy` 추가**. ⚠ `validateDocument`는 unclassified면 ENVELOPE_SCHEMA를 **안 타고** 전용 스키마로 early-return하므로, 봉투에만 넣으면 수집물에는 **전혀 적용되지 않는다** |
| 목록 | `listIds`가 `.meta.json` **배제**(테스트로 고정) | `listSidecarIds()` 신설 (기존 `listIds`·테스트는 불변) |
| 렌더 | 없음 | **`active`일 때만** `rendered/`로 복사(C2-b) |
| 인덱스 | `INDEXED_TYPES` 제외 | `active` 문서만 `rendered/unclassified/index.json`에 컴파일 |
| 미러 | `ALLOWED_TYPE_DIRS`에 없음 | **CLI 선행 릴리스 후** 서버가 내보냄(CS11) |
| 편집 | 슬롯 리졸버 전제 | 슬롯 없음 → `applyEdit`을 타지 않는 본문 PUT |
| **generic 라우트** | `refs`에 있어 가드를 통과 | **명시적 거부**(400 `use_unclassified_api`). 특히 `DELETE`는 현재 존재하지 않는 `<id>.json` 유령 파일을 만들고 진짜 `.md`/`.meta.json`은 그대로 둔다 |
| 감사 | pathspec이 `.json` 고정 | `.md`/`.meta.json` 분기 — 없으면 `/api/audit`가 항상 빈 결과(CS10) |
| 대시보드 | `status === "active"`만 표시(2곳) | `status !== "archived"`로 완화 + 격리 칩 |

json-spec §2 상태 표·§4.5도 함께 개정한다(포맷 진실원이 akg이므로 **spec 버전 범프 +
foundry 통보** 필요).

## 6. 차분 상태 머신

| 조건 | 액션 |
| --- | --- |
| **제안 상태**(C4-b)에 `ref` 없음 · 문서도 없음 | `create` |
| 제안 상태의 `contentHash` 동일 | 없음(no-op) |
| 해시 다름 · 대상 문서 `quarantined` · 사람 편집 없음 | `update` |
| 해시 다름 · 대상 문서 **`active`** | `update-active`(채택 시 강등, C2-c) |
| 해시 다름 · 사람이 본문 편집함 | `conflict`(보고만) |
| `orphan` 후보의 해시 = `create` 후보의 해시 | `rename`(병합, C8) |
| 문서 있음 · `status === "archived"` | `create`(재제안 — 롤백 후 재수집 가능, C4-b) |
| 정형 타입으로 재분류됨 | 슬롯 제안으로 우회(C9) |
| 어댑터 `list()`에 `ref` 없음 | `orphan`(보고만) |

**부분 스캔 안전장치**: 어댑터가 중간에 실패하면 살아있는 문서가 전부 `orphan`으로
보고된다. 따라서 **정상 종료(exit 0) + 완료 마커**가 있을 때만 고아·이동 판정을
수행한다(CS2).

## 9. 보안·운영 규칙 (CS1~CS12)

- **CS1. 어댑터 실행은 신뢰 경계 밖이다.** 화이트리스트(`collectors.json`, 0600, 기동 시
  검증)만으로 부족하다. 이 코드베이스의 spawn 관례는 `env`·`timeout`·`maxBuffer`를 모두
  지정하지 않아 그대로 쓰면 어댑터가 **서버 권한을 상속**한다(users.json·store 접근).
  요구사항: ① 최소 `env` 화이트리스트(`PATH`/`HOME`만) ② 절대경로 명령만 + **실행 파일의
  소유자·퍼미션을 기동 시 검증**(타 사용자 쓰기 가능이면 기동 거부) ③ store 밖 전용 `cwd`
  ④ 벽시계 타임아웃 + `detached` + **프로세스 그룹 kill** ⑤ **스트리밍 파서에서 바이트
  상한 도달 즉시 kill**(다 읽고 거부하면 그 자체가 메모리 DoS) ⑥ stdin 닫기.
- **CS2. 부분 스캔은 고아·이동 판정을 하지 않는다**(§6).
- **CS3. 수집기는 문서를 삭제하지 않는다.** 고아는 보고만.
- **CS4. 입력은 md만.** `mime`이 `text/markdown`이 아니면 항목 거부.
- **CS5. 상한 재산정.** 항목 `text` 256KB / **실행당 항목 200개(C0, 설정 가능)** / 청크
  256KB. 전역 `bodyLimit`(512KB, S13)은 **건드리지 않고** intake 라우트에만 route-level
  limit을 둔다. 항목 초과는 **배치 거부가 아니라 그 항목만 `oversize` 리포트** — v0.1의
  "전부 아니면 전무"는 큰 페이지 하나 때문에 배치 전체가 영원히 죽는 경로였다.
  ("부분 수용 금지"는 **조용한** 부분 수용 금지로 재규정한다.)
- **CS6. 격리 노화 관리.** 30일 미검토는 대시보드 집계, 90일 무활성은 자동 `archived`
  **제안**(자동 실행 아님).
- **CS7. 경로 파라미터는 전부 검증한다.** `runId`·`ref`·`docId` 모두. 그리고
  **`safeJoin`을 "store 루트 밖 금지"에서 "지정 하위트리 밖 금지"로 강화**한다 — 현재
  구현은 store 내부의 하위트리 간 이동을 통과시킨다(§13-1의 실 취약점 근거).
  `store.mjs`의 `validateId`는 현재 **어느 라우트에서도 호출되지 않는 죽은 코드**이므로
  쓰기 진입점에서 실제로 부른다.
- **CS8. 키워드 충돌 검증**(상위 D5가 약속했으나 미구현). 활성화 전에 같은 타입 내 중복은
  **거부**, 타 타입 섀도잉은 **경고**. 근거: 소비 측은 `seenKw`로 **먼저 나온 엔트리가
  키워드를 선점**하고(`keyword-docs.mjs:197`), 그 순서는 `listIds`의 **정렬 없는
  `readdirSync` 순서**(`store.mjs:154`)를 `compileIndex`가 그대로 물려받은 것이다.
  파일명 알파벳순이 **아니다** — 어떤 문서가 키워드를 이기는지를 지금은 **파일시스템이
  정하며, 같은 입력에 같은 결과라는 보장조차 없다**. `compileIndex`에 명시적 정렬 키
  (예: 활성화 시각)를 도입해 결정 근거를 파일시스템에서 뺀다.
- **CS9. 인덱스 재컴파일은 커밋 단위로.** `docWrites`가 문서마다 전량 재컴파일하므로
  N건 일괄 채택이 O(N²) 동기 읽기가 된다(직렬 큐 위에서 전 사용자 쓰기 차단 = S7이
  막으려던 self-DoS). 재컴파일을 `commitFiles` 직전 1회로 올린다.
- **CS10. 감사 추적 가능성.** 커밋 메시지는 `intake <runId> adopt <n> items`, author는
  **채택한 사람**(수집기가 아니라). `runId`로 조회하는 감사 필터를 추가한다 — 없으면
  CS6의 정보가 도달 불가능하다.
- **CS11. 배포 순서.** `rendered/unclassified/`를 서버가 내보내기 **전에** CLI를 먼저
  릴리스한다(`ALLOWED_TYPE_DIRS` 확장). 동시에 `assertKnownTopLevel` 실패를
  **fail-closed(exit 1)**로 전환한다 — 포맷 불일치는 "서버가 죽었다"와 다른 사건인데
  현재는 exit 0이라 **전 사용자 미러가 무증상으로 얼어붙는다**. 계약 테스트로 고정.
- **CS12. 외부 유래 문자열 전량 규율.** 상위 S12는 커밋 메시지 보간을 금지하는데, v0.1의
  CS7은 "본문"만 다뤘다. `reason`·`title`·`ref`도 외부 유래다. 특히 감사 파서가
  `%H\x1f%an\x1f%aI\x1f%s\x1e` 구분자 split이라 `reason`에 리터럴 `\x1f`/`\x1e`를 넣으면
  **가짜 감사 레코드를 위조**할 수 있다. → 제어문자 제거 + 길이 제한, 그리고 감사 파서를
  `-z`/`%x00` 등 **메시지가 담을 수 없는 구분자**로 교체.

## 11. 열린 질문

1. **`quarantined → active` 권한이 approver가 맞나?** C7으로 키워드 부담이 activate로
   몰렸으므로 병목 우려는 커졌지만, 그게 정확히 "희소 슬롯 판단"이라는 게 설계 의도다.
2. **§12-4 재확정**: unclassified를 허브 정식 타입으로 승격(C5의 전제).
3. **위키 md 변환 품질**의 책임은 어댑터(C1)로 정했으나, 변환 실패 문서를 코퍼스에
   넣을지 버릴지의 정책은 미정.
4. **`--direct`를 남길지.** "수집기는 제안만"이라는 규칙에 구멍이지만 격리는 유지된다.
5. **격리 문서의 주입 외 소비 경로**(대시보드 검색·B모드 명시 fetch)를 열지. §0.1대로
   주입 슬롯이 희소하다면, 격리 코퍼스의 가치는 주입이 아니라 **조회 가능성**에 있을 수
   있다 — 그렇다면 그건 수집기가 아니라 akg 제품 방향의 질문이다.

> **v0.1에서 열린 질문이었다가 결정으로 승격된 것**: supersede(→ C4-c, P0),
> `ref` 안정성(→ CS12, 어댑터 계약에 **강제**).

## 12. 상위 설계와의 정합

- §11 S3(프롬프트 주입 채널) → **C2(a·b·c)**: 격리는 주입·배포를 모두 막고, 1회가 아니라
  지속적으로 막는다.
- §D6(사람만 승격) → **C6**: 단, 권한 모델이 자동으로 강제하지 않으므로 스케줄러를

- §D4(어떤 쓰기 경로도 confirmed 직접 생성 불가) → **C2-c**가 그 문서 단위 대응물:

- §D5(키워드 충돌 검증은 서버가) → **CS8**이 그 미구현 약속을 P0으로 끌어온다.

- §12-4 → **C5**가 "편입"으로 확정하며 §5.4 부채를 인수한다.

## 13. 선행 수정 — 수집기와 무관하게 지금 존재하는 결함

수집기 착수 전에 고쳐야 한다. **1번은 재현 완료된 실 취약점이다.**

1. **`proposals.mjs`의 adopt/reject 경로 순회 (치명, 재현됨).** 라우트 파라미터 `pid`를
   검증 없이 경로에 보간한다. Fastify가 `%2F`를 디코드하고(`pid = "../../db-schema/x"`),
   `safeJoin`은 store 루트 밖만 막으므로 하위트리 간 이동이 통과한다. 결과: **approver
   전용인 DELETE를 editor가 우회**해 임의 문서·`rendered/*/index.json`을 삭제할 수 있다.
   실측: 정식 `DELETE` → 403, 순회 `reject` → 200 + 파일 삭제됨.
   → 근본 수정은 CS7의 `safeJoin` 하위트리 강화 + `validateId` 실제 호출.
2. **`spawnSync` `maxBuffer` 미지정 (높음).** 기본 1MB. `rendered/`가 gzip 1MB를 넘으면
   `/api/bundle`이 `ENOBUFS`(`status=null`)로 **영구 500**이고, `r.error`를 검사하지 않아
   원인이 숨는다. `sync`는 fail-open이라 **전 사용자 미러가 조용히 동결**된다.
   코퍼스가 커지면 수집기 없이도 터진다.
3. **`assertKnownTopLevel` 실패가 exit 0 (높음).** 포맷 불일치를 서버 다운과 같이 취급 →
   CS11에서 fail-closed로.
4. **`validateId`가 죽은 코드 (중간).** export만 되고 아무도 호출하지 않는다.
